# 대본에서 최종 영상까지 — AI 영상 제작 플랫폼 Antiframe 개발기

> **개인 프로젝트 · LLM·생성형 AI API · End-to-end Workflow 구현**
> **활동기간: 2026년 5월–현재**

웹툰 숏애니 제작에서 여러 생성 도구를 오가며 같은 정보를 반복 입력하고 작업 상태를 직접 관리해야 했습니다. 이 문제를 개인 프로젝트로 확장해 Gemini·OpenAI·Kling API와 외부 렌더 워커를 연결한 `Antiframe`을 만들었습니다.

## 핵심 요약

- 현업 제작 경험에서 도구 간 단절과 실패 상태 관리 문제를 발견하고 개인 AI 제품으로 확장했습니다.
- 사용자 제작 순서를 먼저 정의한 뒤 여러 API의 구조화된 중간 결과, 검토 지점과 비동기 렌더를 연결했습니다.
- 2026년 6월 8일 준비 입력부터 최종 영상 표시까지 End-to-end 베타 흐름을 확인했습니다.

## 한 편을 만드는 경험에서 반복 작업을 발견했습니다

카카오엔터테인먼트에서 웹툰 컷을 이미지·영상 생성 도구와 TTS, BGM, SFX로 확장해 짧은 애니메이션형 콘텐츠를 제작했습니다. 컷을 고르고 첫 프레임을 정리한 뒤 모션 Prompt를 작성하고 생성 결과를 편집 프로그램에서 연결했습니다.

한 편을 완성하는 과정에서 반복할수록 커지는 문제가 보였습니다.

1. 도구마다 대본과 Prompt를 다시 입력해야 했습니다.
2. 이미지·영상·음성을 장면별로 다시 정리해야 했습니다.
3. 어느 단계가 끝났고 실패한 장면만 어떻게 다시 만들지 확인하기 어려웠습니다.

새로운 생성 모델을 하나 더 쓰는 것만으로 해결되는 문제가 아니었습니다. 제작 단계와 상태, 사람의 검토 지점을 하나의 흐름으로 연결할 필요가 있었습니다. 인턴 실무가 끝난 뒤 이 문제를 개인 AI 영상 제작 플랫폼 `Antiframe`으로 확장했습니다.

## 기능보다 사용자의 제작 순서를 먼저 정했습니다

Antiframe을 다음과 같이 정의했습니다.

> 대본을 넣으면 영상 초안을 만들고 필요한 부분을 검토·수정해 최종 영상까지 완성하는 웹 기반 제작 도구

구현 전에 제품 스펙에서 사용자가 결과를 만들고 확인하는 순서를 먼저 정했습니다.

```text
대본 입력
→ 제목·장면 구성
→ 장면별 내레이션과 화면 자료 생성
→ 사용자 장면 검토와 필요한 부분 수정
→ 움직임·BGM·자막 선택
→ 최종 영상 만들기
→ 결과 확인·다운로드
```

사용자 화면에서는 `storyboard`, `render`, `asset` 같은 내부 용어보다 `영상 초안`, `장면`, `내레이션`, `최종 영상 만들기`를 사용했습니다. 오류도 API 이름보다 어느 단계가 멈췄고 무엇을 다시 할 수 있는지 안내하도록 정리했습니다.

제품 흐름과 정책을 문서로 정의하고 구현 뒤 화면과 동작이 스펙과 다르면 코드를 다시 맞추는 방식으로 작업했습니다.

![Antiframe에서 영상 형식과 제작 단계를 선택하는 화면](assets/screenshots/antiframe-format-select.png "대본 입력에 앞서 영상 형식과 전체 제작 단계를 한 화면에서 확인하도록 구성했습니다.")

## 여러 AI API를 단계별 역할에 맞게 연결했습니다

한 모델에 모든 작업을 맡기지 않고 제작 단계의 목적에 맞는 서비스를 연결했습니다.

| 제작 단계 | 사용 서비스 | 역할 |
|---|---|---|
| 대본·제목·장면 구성 | Gemini API | 한국어 대본을 장면별 구조화 데이터로 변환 |
| 내레이션 | OpenAI TTS | 장면별 음성 생성 |
| 이미지 | OpenAI Image | 필요한 장면 이미지 생성 |
| 스톡 자료 | Pexels API | 대본에 맞는 이미지·영상 후보 검색 |
| 장면 모션 | Kling API | 이미지에서 짧은 영상 클립 생성 |
| 프로젝트·작업 상태 | Supabase | 사용자·프로젝트·토큰·렌더 큐 저장 |
| 최종 영상 | Mac mini·After Effects | 장면·음성·자막·BGM을 MP4로 렌더 |

Gemini 응답은 화면에 바로 노출하지 않고 장면 번호, 내레이션, 장면 설명과 검색어처럼 후속 단계가 사용할 수 있는 구조로 변환했습니다. 생성 결과는 프로젝트·장면 단위로 저장해 새로고침 뒤에도 다시 열 수 있게 했습니다.

유료 API 호출은 환경변수로 명시적으로 켜고, 기본 검증에서는 시뮬레이션 결과로 화면과 상태 전환을 확인하도록 분리했습니다. 제품 흐름 테스트와 비용이 발생하는 생성을 구분해 반복 검증할 수 있게 한 선택이었습니다.

## 자동 생성보다 검토와 복구가 가능한 상태를 만들었습니다

각 단계의 결과를 바로 다음 단계에 확정값으로 넘기지 않았습니다. 사용자는 장면 구성과 자료 후보를 확인하고 필요한 부분만 다시 생성할 수 있습니다. 완전 자동화보다 수정 비용이 낮은 중간 결과에서 사람이 방향을 결정하도록 했습니다.

실패한 작업도 삭제하지 않고 상태로 남겼습니다. 진행 중인 렌더가 있을 때 중복 요청을 막고, 실패 또는 완료 뒤에는 같은 프로젝트를 다시 렌더링할 수 있게 했습니다.

이는 사람이 마지막 결과만 확인하는 구조와 다릅니다. 어느 단계에서 결과가 만들어졌고 어디에서 실패했는지를 알 수 있어야 필요한 단계부터 복구할 수 있습니다.

## 웹 밖의 렌더링까지 하나의 흐름으로 연결했습니다

이미지와 음성이 생성돼도 영상은 완성되지 않습니다. 장면 길이를 내레이션에 맞추고 자막과 BGM을 합쳐 MP4로 렌더링해야 합니다. 무거운 편집 작업은 웹 서버가 아니라 외부 워커가 처리하도록 구성했습니다.

> **[시각자료 07 삽입 위치]**
> 대본 구조화·생성 API·사용자 검토·렌더 큐와 외부 워커·최종 결과를 다섯 단계로 묶은 단순 시스템 흐름도

```text
웹에서 최종 영상 요청
→ Supabase render_jobs에 queued 저장
→ Mac mini 워커가 작업 claim
→ After Effects 렌더
→ 50MB 초과 시 ffmpeg 압축
→ Supabase Storage 업로드
→ 웹 결과 화면에 최종 영상 표시
```

초기에는 파일 크기 제한과 ffmpeg 실행 경로 때문에 업로드가 멈췄습니다. 목표 용량을 50MB 이하로 정하고 저장소의 `ffmpeg-static`을 사용하는 대체 경로를 추가했습니다. 2026년 6월 8일 준비 입력부터 최종 영상 표시까지 베타 흐름이 끝까지 동작하는 상태를 확인했습니다.

## 운영 판단에 필요한 자료도 함께 만들었습니다

개인 MVP라도 다른 사람이 범위와 실패 지점을 이해할 수 있어야 서비스로 검토할 수 있다고 보았습니다.

- 로그인, 프로젝트 저장과 다시 열기
- 장면별 산출물과 진행 상태 관리
- 생성 비용을 구분하기 위한 토큰 차감 구조
- 실패 단계 재시도와 중복 작업 방지
- 관리자 화면과 대기 신청 관리
- 제품 스펙, API 연동 명세, 운영 노트와 변경 기록
- 반복 UI·상태·오류 처리 기준을 모은 Template Library

이 자료는 생성 호출, 상태 관리, 외부 렌더와 결과 복구가 하나의 제품 흐름에서 작동하는지를 팀원이 다시 확인할 수 있는 운영 기반입니다.

## 베타 구현 결과와 다음 단계

대본 입력, AI 기반 장면 구성, 음성·이미지·영상 생성, 사용자 검토, 외부 렌더 워커와 결과 저장을 연결했습니다. 한 기능의 데모를 넘어 작업 상태, 실패·재시도와 운영 문서가 포함된 베타 제작 흐름을 확인했습니다.

현재는 End-to-end 베타 제작 흐름을 기준으로 동작합니다. 다음 단계에서는 실제 결제와 장기 운영용 DB·Storage를 연결하고, 같은 입력 세트로 모델별 생성 품질·비용·지연시간을 비교해 운영 기준을 정교화할 계획입니다.

## 기업용 AI Workflow로 확장할 수 있는 구조

Antiframe에서 만든 `여러 API의 순차 작업 → 구조화된 중간 결과 → 사람 검토 → 비동기 Job → 실패 기록과 재시도` 구조는 단계가 많은 기업용 AI Workflow에도 적용할 수 있습니다.

문서 검토, Desktop Research와 보고서 작성처럼 여러 단계가 연결된 기업용 AI Workflow에서도 각 단계의 결과를 저장하고 담당자가 승인할 지점을 구분할 수 있습니다. 실제 적용에서는 현업 전문가가 입력 데이터, 승인 기준과 최종 책임을 먼저 정해야 합니다.

## 이 경험이 남긴 기준

Antiframe은 생성형 AI를 많이 연결한 프로젝트보다, 현장에서 발견한 반복 작업을 통제·복구 가능한 End-to-end Workflow로 발전시킨 Case입니다.

문제를 직접 경험한 뒤 제품 흐름, API 연동, 외부 렌더와 운영 문서까지 구현한 과정은 상상에 머무르지 않고 작동하는 결과로 확인하는 제 실행 방식을 가장 분명하게 보여줍니다.
