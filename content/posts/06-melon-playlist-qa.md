# 47개 페르소나로 대화형 AI 플레이리스트를 검수하기

> **카카오엔터테인먼트 인턴 실무 · AI 서비스 QA · 사용자 시나리오 설계**

자연어로 플레이리스트를 만드는 대화형 AI QA에서 서로 다른 생활 리듬과 서비스 경험을 반영한 47개 페르소나를 구성했습니다. 사용자가 처음 말한 조건이 추천 결과, 후속 대화와 TTS까지 유지되는지를 실제 이용 흐름으로 검수했습니다.

## 핵심 요약

- 평균적인 주간 직장인만으로 발견하기 어려운 조건을 찾기 위해 생활시간·활동·취향·서비스 숙련도를 조합했습니다.
- 추천곡이 개인적으로 마음에 드는지보다 사용자가 명시한 조건과 제외 요청의 반영 여부를 기준으로 기록했습니다.
- 추천 모델을 개발한 경험이 아니라 사용자 관점의 질문·기대 결과·Pass/Fail을 설계하고 전달한 QA 경험입니다.

> **[시각자료 16 삽입 위치]**  
> 새벽 배송·재택근무·운동·육아 등 서로 다른 상황에서 음악을 요청하는 사용자들의 대표 이미지

---

## 프로젝트 소개: 한 문장의 요청이 플레이리스트가 되는 과정

대화형 음악 추천은 사용자가 정확한 곡명을 몰라도 상황과 기분을 설명해 플레이리스트를 만들 수 있는 기능입니다.

```text
새벽 배송 중에 졸리지 않으면서
오래 들어도 피곤하지 않은 음악을 추천해줘.
```

사용자는 듣는 시간, 활동, 기분, 선호 아티스트, 원하는 시대와 제외 조건을 자연어로 입력합니다. 첫 결과 뒤에는 “조금 덜 신나게”, “가사 없는 곡 위주로”처럼 조건을 수정할 수 있습니다.

QA에서는 플레이리스트가 화면에 생성되는지만 보지 않았습니다. 사용자의 조건이 추천곡과 설명에 반영됐는지, 후속 대화에서 앞선 조건을 유지하는지, 추천 설명을 TTS가 알아들을 수 있게 읽는지까지 확인했습니다.

## 제가 담당한 범위

```text
사용자의 자연어·선택지 입력
→ 요청 조건 해석
→ 플레이리스트와 추천 설명 생성
→ 후속 대화에서 조건 수정·유지
→ 추천 문장의 TTS 출력
```

저는 추천 모델이나 대화 시스템을 개발하지 않았습니다. 사용자 페르소나, 질문과 기대 조건을 만들고 실제 결과를 검수해 담당자에게 공유했습니다.

## 평균적인 사용자보다 조건의 조합을 넓혔습니다

초기 추천 예시에는 일반적인 주간 직장인의 생활시간이 많이 반영돼 있었습니다. 실제 이용자는 야간·교대근무를 하거나 재택근무, 운동과 육아 중에 음악을 듣기도 합니다. 플레이리스트 생성에 익숙하지 않아 모호한 일상어만 쓰는 사용자도 있습니다.

다음 축을 조합해 47개 페르소나를 만들었습니다.

| 페르소나 축 | 질문에 미치는 영향 |
|---|---|
| 생활시간·근무 형태 | 아침·심야·교대근무 등 사용 시점 |
| 활동·목적 | 집중·운전·운동·휴식 등 필요한 자극 수준 |
| 선호·제외 조건 | 장르·시대·아티스트와 피하고 싶은 요소 |
| 서비스 숙련도 | 구체적인 조건 입력 또는 모호한 일상 표현 |
| 후속 대화 방식 | 조건 추가·완화·제외와 이전 맥락 유지 |

> **[시각자료 17 삽입 위치]**  
> 생활시간·활동·선호·숙련도·대화 방식의 축을 조합해 47개 페르소나를 만드는 구성도

숫자를 채우기 위해 이름과 나이만 바꾸지 않았습니다. 같은 음악 취향이어도 듣는 시간과 활동이 다르면 적절한 추천 조건도 달라지도록 상황을 설계했습니다.

## 취향보다 사용자가 말한 조건을 기준으로 봤습니다

```text
첫 요청
새벽 배송하면서 졸리지 않게 들을 플리 만들어줘.

후속 요청
너무 시끄러운 곡은 빼줘. 오래 들어도 피곤하지 않았으면 좋겠어.

확인 조건
- 새벽 배송과 집중 유지 맥락이 남아 있는가
- 높은 자극 제외 요청이 최종 결과에 반영되는가
- 두 조건이 서로 충돌할 때 설명하거나 조정하는가
```

추천곡이 개인적으로 마음에 드는지는 재현 가능한 기준이 되기 어렵습니다. Google Sheet에 페르소나, 입력, 실제 답변, 기대 조건과 Pass/Fail을 기록하고 사용자가 직접 말한 시대, 활동, 분위기와 제외 조건의 반영 여부를 확인했습니다.

자연스러운 설명이 나와도 명시 조건을 놓치면 Fail로 보았습니다. 반대로 제가 평소 듣지 않는 곡이어도 조건에 부합한다면 취향을 이유로 Fail 처리하지 않았습니다.

## 후속 대화에서는 새 조건과 이전 조건을 함께 확인했습니다

첫 답변만 적절해도 후속 요청에서 이전 맥락을 잃으면 사용자는 조건을 처음부터 다시 입력해야 합니다.

> **[시각자료 18 삽입 위치]**  
> 첫 요청의 시간·활동 조건에 후속 요청의 자극 제외 조건이 더해져 최종 추천까지 유지되는 대화 흐름도

후속 대화에서는 다음 세 가지를 구분했습니다.

1. 유지해야 할 기존 조건
2. 새로 추가된 조건
3. 사용자가 명시적으로 바꾸거나 제거한 조건

예를 들어 “조금 덜 신나게”는 새 플레이리스트를 처음부터 만드는 요청이 아니라 현재 결과의 자극 수준만 조정하는 요청입니다. AI가 이전의 운동·시대 조건까지 버리는지 함께 확인했습니다.

## 사용자 표현과 TTS까지 확인했습니다

사용자는 정해진 분류명만 쓰지 않습니다. 아티스트 줄임말, “2000년대 감성”, “너무 세지 않게”처럼 분류표에 바로 대응하기 어려운 표현도 테스트했습니다.

텍스트 답변이 적절해도 곡명과 아티스트명을 TTS가 잘못 읽거나 의미 단위와 다른 위치에서 쉬면 최종 경험이 달라졌습니다. 발음 오류는 발음사전 후보로, 문장 호흡 문제는 끊어읽기 이슈로 구분했습니다. 발음사전 데이터 구축은 별도 게시글에서 다룹니다.

## 적용 결과와 한계

47개 페르소나를 통해 생활시간, 활동, 서비스 숙련도와 자연어 표현의 범위를 넓혀 추천 결과·후속 대화·TTS를 검수했습니다. 기대 조건과 실제 결과를 남겨 담당자가 같은 입력을 다시 확인할 수 있게 했습니다.

제가 추천 로직을 개발하거나 최종 서비스 정책을 결정한 것은 아닙니다. 전체 테스트 통과율, 추천 품질의 개선 폭과 실제 사용자 만족도도 측정하지 않았습니다. 내부 입력·화면·결과는 공개할 수 없어 이 글의 사례는 문제 구조를 유지한 공개용 예시로 재구성했습니다.

## 이 경험이 남긴 기준

AI 서비스의 품질은 평균적인 한 번의 답변만으로 판단하기 어렵습니다. 서로 다른 사용 환경, 이전 대화의 조건과 최종 출력 채널까지 이어서 확인해야 합니다.

47개 페르소나는 다양한 사용자를 상상하는 데서 끝나지 않고, 각 사용자의 언어를 테스트 입력과 기대 결과로 바꿔 직접 검증한 경험입니다.
