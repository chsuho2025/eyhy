# 팬 활동 데이터에서 지금 대화에 필요한 정보만 남기는 법

> **카카오엔터테인먼트 사내 AI 해커톤 · Gemini API · 팀 최종 3위·우수상**

팬의 최근 공개 활동을 Gemini API로 정리해 아티스트의 답글 작성을 돕는 프로토타입을 팀으로 제작했습니다. 모든 활동을 요약하는 대신 이번 대화에 필요한 입력만 남기고, 근거 밖의 해석과 실행할 수 없는 약속을 제한했습니다.

## 핵심 요약

- 팬의 과거 활동은 개인화된 답글에 도움이 될 수 있지만 무관한 기록까지 사용하면 과잉 추론으로 이어질 수 있었습니다.
- 입력 정제, 근거 안의 구조화 출력, 위험 활동 분리와 아티스트 최종 확인을 사용 흐름에 반영했습니다.
- 프로토타입과 수상 결과는 확인했지만 실제 팬 데이터의 품질·개인정보 탐지율과 아티스트 활용성은 측정하지 않았습니다.

---

## 프로젝트 소개: 관계를 대신하지 않고 대화를 돕는 AI

팬 플랫폼에는 한 팬의 게시글, 댓글과 콘텐츠 반응이 계속 쌓입니다. 아티스트가 지금 눈앞의 댓글에 답할 때 팬이 이전에 어떤 콘텐츠를 좋아했고 무엇을 질문했는지 알 수 있다면 대화를 이어가는 데 도움이 될 수 있습니다. 하지만 답글을 작성할 때마다 과거 활동을 모두 다시 읽기는 어렵습니다.

해커톤 팀은 최근 공개 활동을 생성형 AI가 정리해 아티스트에게 답글 참고 정보를 보여주는 프로토타입을 기획했습니다. 기능의 목적은 AI가 아티스트 대신 팬과 관계를 관리하는 것이 아니라, 많은 기록 가운데 이번 대화에 필요한 단서를 짧게 찾도록 돕는 것이었습니다.

이를 위해 다음 세 질문을 먼저 정했습니다.

1. 지금 답글에 필요한 데이터는 무엇인가
2. 모델은 제공된 기록에서 어디까지 표현할 수 있는가
3. 생성 결과의 최종 사용 여부는 누가 결정하는가

## 제가 참여한 범위

저는 문제 정의와 사용자 흐름, 입력 활동의 정제 기준, Prompt 조건, 안전 기준과 출력 구조를 검토했습니다.

- 답글과 관련된 최근 활동 범위 검토
- 중복·저정보 활동 정제 기준
- 제공 기록 밖의 감정·관계 추론 제한
- 위험 활동과 정상 답글 생성 분리
- 구조화 출력과 아티스트 확인 단계

팀 전체의 구현·수상 결과와 제 기여를 혼동하지 않도록, 이 글에서는 제가 참여한 문제 정의·입력 정제·Prompt 및 안전 기준 검토 범위만 설명합니다.

## API 호출 전에 입력부터 줄였습니다

초기에는 최근 활동을 한 번에 Gemini API에 전달하는 방식을 생각했습니다. 그러나 입력할 수 있는 데이터와 이번 답글에 필요한 데이터는 달랐습니다.

- 이미 확인한 중복 내용
- `ㅋㅋㅋㅋ`, `좋아요`처럼 구체적인 관심사를 확인하기 어려운 반복 반응
- 현재 답글과 연결하기 어려운 오래된 활동
- 위협·성적 표현 등 정상 답글과 분리해야 하는 위험 가능 활동

```text
최근 공개 활동
→ 중복·저정보 활동 정제
→ 위험 가능 활동 분리
→ 현재 답글에 필요한 기록만 Gemini API에 전달
```

> **[시각자료 03 삽입 위치]**
> 전체 팬 활동이 중복·저정보 제거와 위험 분리를 거쳐 최소 입력이 되고 아티스트 확인으로 이어지는 단순 데이터 파이프라인

이 전처리는 토큰을 줄이는 것만을 의미하지 않았습니다. 모델이 사용할 수 있는 근거의 범위를 정해 불필요한 데이터 사용과 과잉 해석 가능성을 줄이기 위한 설계였습니다.

## 구조화된 출력과 내용의 적절성을 분리했습니다

Gemini API 결과는 자유로운 장문 대신 다음 항목으로 나누었습니다.

- 지금 답글에 참고할 최근 관심사
- 팬이 남긴 질문이나 구체적인 반응
- 안전 여부
- 답글 참고 문장

Prompt에는 제공된 활동에서 직접 확인되는 사건과 대상만 사용하고 기록에 없는 감정·관계·시점을 만들지 않도록 조건을 두었습니다.

```text
입력
요즘 이 노래를 자주 듣고 있어요.

위험한 확장
힘든 시기를 이 노래로 위로받고 있습니다.

기대 방향
최근 해당 곡을 자주 듣는다고 언급했습니다.
```

형식에 맞는 JSON이 생성됐다는 사실과 내용이 적절하다는 판단은 달랐습니다. 구조 준수와 입력 근거의 일치를 별도의 확인 대상으로 두었습니다.

## 친근한 답글과 실행할 수 있는 답글은 달랐습니다

팬이 “다음 행사에서도 꼭 만나요”라고 썼다고 해서 실제 만남, 좌석이나 일정을 보장하는 문장을 제안하면 아티스트가 지킬 수 없는 약속이 됩니다.

```text
위험한 제안
다음 행사에서 꼭 만날 수 있게 준비할게요.

기대 방향
다음 활동도 기대해 줘서 고마워요.
```

위협이나 심한 괴롭힘이 포함된 활동도 일반적인 관심사처럼 요약해 친근한 답글을 제안하지 않도록 안전 여부와 정상 답글 생성을 분리했습니다.

최종 결과는 팬에게 자동 전송하지 않았습니다. 아티스트가 요약과 참고 문장을 확인하고 실제 사용할지 결정하도록 했습니다. 사람의 최종 확인만으로 오류가 사라지는 것은 아니므로 그 앞단의 입력 범위와 생성 조건도 함께 제한했습니다.

## 결과와 한계

팀은 제한된 해커톤 기간 안에 Gemini API 기반 프로토타입을 완성했고 최종 3위로 우수상을 받았습니다. 입력 정제, 구조화 출력과 사용자 확인이 이어지는 흐름을 작동 가능한 결과물로 만들었습니다.

수상 결과는 서비스 품질 검증과 같지 않습니다. 실제 팬 데이터에서 개인정보 탐지율을 측정하지 않았고 여러 언어권의 요약 품질과 아티스트 활용성도 확인하지 못했습니다. 당시 공식 대규모 평가표를 운영한 것도 아닙니다.

해커톤 이후 같은 기능을 검토할 경우 필요한 근거 일치, 관련성, 안전 판정, 실행 가능성, 정보 충분성과 형식 준수 항목을 정리했습니다. 이는 당시 구현·수상 결과와 구분되는 사후 평가 설계입니다.

## 다른 지식 업무에 적용한다면

팬 활동과 전문 문서는 데이터의 성격과 위험이 다릅니다. 그러나 `필요한 데이터만 선별 → 제공 근거 안에서 구조화 → 위험 가능 항목 분리 → 담당자 최종 확인`이라는 흐름은 문서·규정·내부 보고서의 요약 보조에도 적용할 수 있습니다.

이 경험은 LLM API 호출보다 먼저 입력 데이터, 허용할 출력과 사람의 최종 판단을 함께 설계해야 한다는 기준을 남겼습니다.
