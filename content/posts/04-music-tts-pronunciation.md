# 멜론 AI가 음악 고유명사를 정확히 읽기 위한 TTS 발음사전

> **카카오엔터테인먼트 인턴 실무 · 데이터 전처리·가공 · TTS 품질 검수**

영어·숫자가 포함된 곡명과 아티스트명 약 3만 5천 개를 발음사전 후보로 정리하고, 초기 약 3,500건을 공개 자료로 조사했습니다. 발음만 입력하지 않고 근거·예외·처리 시간을 함께 기록해 반복 가능한 항목과 사람의 판단이 필요한 항목을 구분했습니다.

## 핵심 요약

- 화면에서 정확한 음악 고유명사도 TTS로 들으면 알아듣기 어려울 수 있어 음악 도메인 발음 데이터를 별도로 구축했습니다.
- 공개 자료의 출처와 판단 이유를 남기고, 국적·장르·표기 구조별 반복 패턴과 예외를 구분했습니다.
- 인턴 당시 Python이나 LLM API로 자동화한 것은 아니며, 수동 데이터에서 자동화 후보와 사람 검토 경계를 발견한 경험입니다.

---

## 프로젝트 소개: 보이는 이름과 들리는 이름은 달랐습니다

음악 추천 문장에는 곡명, 아티스트명과 클래식 작품 번호가 반복해서 등장합니다. 화면에서는 정확하게 표기된 이름도 TTS가 읽으면 사용자가 어떤 음악인지 알아듣기 어려운 소리로 바뀔 수 있습니다.

음악 고유명사에는 여러 언어와 숫자, 기호가 섞여 있습니다. 국내에서 널리 사용하는 발음이 원어를 기계적으로 읽은 결과와 다르기도 하고, 숫자가 철자가 되거나 아티스트가 직접 정한 조어인 경우도 있습니다.

제가 참여한 업무는 사내 TTS에 사용할 음악 도메인 발음사전을 정리하고, 실제 추천 문장에 적용된 결과를 검수하는 일이었습니다. 일부 발음 데이터는 대화형 음악 추천 기능의 설명 문장을 읽는 TTS에도 적용됐습니다.

## 수동 조사를 다시 확인할 수 있는 데이터로 남겼습니다

영어와 숫자가 포함된 약 3만 5천 개를 후보로 두고, 초기 약 10%인 3,500건가량을 직접 조사했습니다. 처음부터 자동 분류 결과를 검수한 것이 아니라 공개 자료를 검색해 하나씩 확인한 작업이었습니다.

각 항목에는 발음 결과 외에 다음 정보를 함께 기록했습니다.

| 기록 항목 | 남긴 이유 |
|---|---|
| 검색어·확인 자료 | 다른 사람이 판단 근거를 다시 확인하기 위해 |
| 국적·활동 정보 | 우선 확인할 언어권과 통용 표기의 단서를 얻기 위해 |
| 국내 통용 표기 | 서비스 사용자가 익숙한 발음과 원어 발음의 차이를 보기 위해 |
| 선택 이유·예외 조건 | 같은 형태의 이름을 일관되게 처리하기 위해 |
| 항목별 처리 시간 | 자동화가 필요한 병목과 어려운 유형을 찾기 위해 |

발음만 저장하면 결과가 틀렸을 때 어느 단계에서 다시 확인해야 하는지 알기 어렵습니다. 출처와 판단 이유를 함께 남겨야 다음 업데이트와 검수에 사용할 수 있는 데이터가 된다고 보았습니다.

> **[시각자료 04 삽입 위치]**
> 약 35,000개 후보에서 3,500건을 조사한 뒤 반복 패턴은 자동화 후보로, 예외는 사람 검토로 나누는 데이터 처리 흐름도

## 개별 이름에서 반복되는 패턴을 찾았습니다

처음에는 모든 이름이 서로 다른 예외처럼 보였습니다. 기록이 쌓이자 조사 순서를 줄일 수 있는 단서가 나타났습니다.

- 국적: 우선 확인할 언어권 후보
- 장르: 숫자와 알파벳을 사용하는 관용 표현
- 표기 구조: `Op.`, 곡 번호, 조성과 작품목록 번호
- 국내 통용명: 원어 발음과 국내 서비스에서 익숙한 이름의 차이

반면 `6LACK`, `deadmau5`, `Tech N9ne`처럼 숫자와 철자를 의도적으로 조합한 이름은 겉모양만으로 읽는 방식을 확정하기 어려웠습니다. 같은 숫자도 이름마다 읽는 방식이 달라 일반 규칙을 일괄 적용하면 오히려 오류를 늘릴 수 있었습니다.

따라서 패턴을 정답을 자동 결정하는 규칙이 아니라, 사람이 먼저 확인할 범위를 줄이고 일관된 발음 후보를 제안하는 보조 기준으로 사용했습니다.

## 자동화할 수 있는 일과 사람이 확인해야 할 일을 나누었습니다

| 데이터 유형 | 적절한 처리 방향 |
|---|---|
| 반복되는 클래식 표기 | 규칙으로 발음 후보 생성 |
| 숫자·알파벳 관용 표현 | 장르·표기 조건으로 후보 분류 |
| 공식 발음·국내 통용명이 확인되는 항목 | 근거와 함께 사전 등록 |
| 조어·밈·고유한 아티스트명 | 공개 자료 확인과 수동 검토 유지 |
| 출처가 서로 다른 항목 | 자동 확정하지 않고 검토 대상으로 표시 |

지금 다시 만든다면 Python으로 표기 구조와 메타데이터를 전처리하고, LLM API에는 근거 후보의 요약과 검토 우선순위 지정을 맡길 수 있습니다. 다만 모델이 만든 발음을 그대로 등록하지 않고 공식 인터뷰, 국내 유통 표기와 실제 TTS 음성을 사람이 확인해야 합니다.

이는 당시 구현 결과가 아니라 수동 조사 기록에서 도출한 후속 자동화 구상입니다. 인턴 당시 제가 Python 또는 LLM API 자동화 프로그램을 개발했다고 표현하지 않습니다.

## 단어가 아니라 실제 사용 문장으로 검수했습니다

발음사전은 단어 단위로 등록하지만 사용자는 완성된 추천 문장 안에서 이름을 듣습니다. 단독 발음이 맞아도 조사와 붙을 때 소리가 달라지거나 문장 중간의 호흡이 어색하면 알아듣기 어려웠습니다.

```text
검수 항목: BWV 846
테스트 문장: 바흐의 BWV 846을 차분한 아침 음악으로 골랐어요.

확인 항목
- 알파벳 약어와 작품 번호를 의도한 방식으로 읽는가
- 조사 앞에서 부자연스럽게 끊지 않는가
- 등록한 발음사전이 실제 문장에 적용됐는가
```

주 단위 TTS 업데이트 뒤 같은 문장을 재생해 적용 여부를 확인했습니다. 고유명사를 잘못 읽으면 발음사전 이슈로, 개별 단어는 맞지만 의미 단위와 다른 곳에서 쉬면 TTS 호흡 이슈로 구분했습니다.

## 적용 결과와 한계

약 3,500건의 발음, 확인 근거와 처리 시간을 기록하고 국적·장르·표기 구조에 따른 반복 패턴과 예외를 분류했습니다. 등록된 항목은 추천 문장으로 다시 재생해 적용 여부를 확인했고 일부 발음사전의 서비스 적용도 확인했습니다.

다만 적용 전후 오독률이나 사용자 이해도는 측정하지 못했습니다. 제가 TTS 모델을 개발하거나 최종 발음 정책을 결정한 것도 아닙니다. 공개 자료마다 발음이 다른 항목은 담당자와 함께 판단해야 했습니다.

## 이 경험이 남긴 기준

AI가 학습하거나 참조하는 데이터는 양뿐 아니라 출처, 판단 이유와 예외를 함께 관리해야 합니다. 자동화는 확인 가능한 반복 작업부터 시작하고, 고유한 맥락과 출처 충돌이 있는 데이터에는 사람의 검토를 남겨야 합니다.

수동으로 시작한 발음 조사를 근거가 있는 데이터와 자동화 가능한 단위로 바꾼 경험은 문서 분류, 지식 데이터 구축과 AI 학습용 데이터 가공에서도 사용할 수 있는 기준이 됐습니다.
